# Project Structure

How the My Mechanic QLD Next.js project is organised, how data flows through it, how pages render, and how to extend it. Read this first if you are coming to the codebase cold.

---

```
MyMechanicQLD/

├── components/                   Reusable React components
```

### Dynamic / templated pages (one file, many URLs)

| URL pattern | File | Content source |
|---|---|---|
| `/{service-slug}/` | `app/[slug]/page.tsx` | `content/services/{slug}.mdx` |
| `/blog/{post-slug}/` | `app/blog/[slug]/page.tsx` | `content/posts/{slug}.mdx` |

The catch-all `app/[slug]/page.tsx` serves: `/brake-repairs/`, `/starter-alternator/`, `/radiator-cooling-system/`, `/logbook-servicing/`, `/pre-purchase-inspection/`, `/battery-replacement/`, `/car-diagnostics/`, `/steering-suspension/`. Static routes (everything with its own folder under `app/`) take priority — so `/services/` always hits `app/services/page.tsx`, never the catch-all.

## How a page renders

The flow for any URL on the site:

```
Browser request
      ↓
Vercel (edge) checks redirects.json — 301 if matched
      ↓
Next.js matches the URL to a file in app/
      ↓
For dynamic routes (services, blog posts):
  → lib/services.ts or lib/posts.ts reads the MDX file
  → gray-matter parses YAML frontmatter
  → remark + remark-gfm + remark-html converts body to HTML
  → React server-renders the page with frontmatter + bodyHtml
      ↓
Page rendered with:
  - <html> with locale and font variables
  - <head> with all <meta> tags (from Next.js Metadata API)
  - <body> with site-wide LocalBusiness JSON-LD + WebSite JSON-LD
  - <Header /> with utility bar + nav + mobile drawer
  - <main> with the page content + per-page JSON-LD
  - <Footer />
      ↓
HTML streamed to browser
```

Server components are the default. The only client component on most pages is `MobileDrawer.tsx` (needs state for open/closed) and `CoverageChecker.tsx` (needs state for the form result).

Per-page components (the body of `{children}`):

- Sections (`<section>`) typically have responsive padding `py-14 md:py-20 lg:py-24`
- Each section wraps content in `<div className="container">` which is responsive (1rem on mobile, scales up)
- Service pages, blog posts, FAQ pages and others inject their own JSON-LD inline via `<script type="application/ld+json">`

## Data flow for MDX-driven pages

### Service pages

```
content/services/brake-repairs.mdx
  ↓ (frontmatter has slug, title, hero, FAQ, pricing, etc.)
lib/services.ts
  ↓ getService('brake-repairs')
  ↓ getAllServices()
  ↓ getServiceSlugs()
app/[slug]/page.tsx
  ↓ generateStaticParams() lists all slugs at build
  ↓ generateMetadata() builds title/description/OG from frontmatter
  ↓ default export renders:
       - Breadcrumb
       - Hero with image
       - Spec strip (price, duration, warranty)
       - Body HTML
       - Parts brands strip
       - FAQ accordion
       - Related services grid
       - CTA
       - 3 JSON-LD blocks (Service + FAQPage + BreadcrumbList)
```

### Blog posts

```
content/posts/how-much-does-a-brake-job-cost-in-brisbane.mdx
  ↓ (frontmatter has slug, title, author, dates, hero, category, related)
lib/posts.ts
  ↓ getPost('how-much-...')
  ↓ getAllPosts() — sorted by date desc
  ↓ getRelatedPosts(post) — explicit related, then same category, then newest
app/blog/[slug]/page.tsx
  ↓ generateStaticParams() lists every post slug
  ↓ generateMetadata() with article-type OG
  ↓ default export renders:
       - Breadcrumb
       - Article header (category, title, author byline, dates, reading time)
       - Hero image (16:9)
       - Body HTML in .prose-mmq
       - Author bio card
       - Related services grid (from post.related_services)
       - Related posts grid (from getRelatedPosts)
       - CTA
       - 2 JSON-LD blocks (BlogPosting + BreadcrumbList)
```

## Metadata / SEO strategy

### Defaults (in `app/layout.tsx`)

The root layout exports `metadata: Metadata` with:

- Site-wide title default and template
- Default description (symptom-led)
- 47 keywords across 7 intent categories
- Authors, creator, publisher, category
- `formatDetection: { telephone: true, address: true, email: true }`
- Open Graph defaults (en_AU, site name, default image)
- Twitter Card defaults
- robots config with explicit googleBot directives
- Icons and favicons

### Per-page overrides

Every page exports its own `metadata` (or `generateMetadata` for dynamic routes) that overrides the defaults with page-specific values. Examples:

- `/about/page.tsx` exports `metadata: { title: 'About My Mechanic QLD | ...', description: '...', alternates: { canonical: '/about/' }, openGraph: {...} }`
- `app/[slug]/page.tsx` exports `generateMetadata` that reads each service's frontmatter

### Structured data (JSON-LD)

Three layers:

1. **Site-wide** (in `app/layout.tsx`): `LocalBusiness` (with address, geo, service area, hours, offer catalog) + `WebSite` with `SearchAction`.
2. **Page-type** (in each page's component): `Service` + `FAQPage` + `BreadcrumbList` on services. `BlogPosting` + `BreadcrumbList` on blog posts. `HowTo` on `/how-it-works/`. `ItemList` on `/services/` and `/case-studies/`. `FAQPage` on `/faq/`, `/pricing/`, `/warranty/`, `/how-it-works/`.
3. **AI-readable surfaces**: `/llms.txt` and `/llms-full.txt` in `public/`. Read by ChatGPT, Claude, Perplexity and others.

## Forms and server actions

There is one server action: `submitQuoteAction` in `app/actions.ts`. It:

1. Receives `FormData` from the quote form
2. Logs the submission (TODO: wire to Resend / Formspree before launch)
3. Redirects to `/?submitted=true#quote` or `/contact/?submitted=true` or `/book/?submitted=true`

The receiving page reads `searchParams.submitted` and renders a success state instead of the form.

The same `<QuoteForm />` component is used on three pages: `/`, `/contact/`, `/book/`. It is a server component that submits to the action. No client JS for the form itself.

## How to add a new service page

1. Create `content/services/your-new-service.mdx`
2. Copy the frontmatter shape from any existing service file (slug, title, meta_description, hero, FAQ, related_services, breadcrumb, etc.)
3. Write the body in markdown
4. Done. The page is live at `/your-new-service/` next time the app runs.

The `app/[slug]/page.tsx` template auto-discovers the file via `getServiceSlugs()` and renders it with all the schema markup. The sitemap auto-includes it on next build. No code changes needed.

---

## How to add a new blog post

Same flow as a service page, but:

1. Create `content/posts/your-post-slug.mdx`
2. Use the frontmatter shape from any existing post (slug, title, meta_description, excerpt, category, tags, author, date_published, hero_image, related, related_services)
3. Write the body in markdown
4. The post appears at `/blog/your-post-slug/`, in the blog index sorted by date, and in the sitemap.

## How to add a new top-level static page

```
app/new-page/page.tsx
```

Export `metadata` (for SEO) and a default function that returns JSX. Optionally add the URL to `app/sitemap.ts`. Footer / nav link adjustments live in `components/Footer.tsx` and `components/Header.tsx`.

## Where the magic actually happens

If you only read three files to understand the codebase, read:

1. **`app/layout.tsx`** — the site-wide metadata, schema and frame
2. **`lib/services.ts`** — how MDX becomes a rendered page (the same pattern as `lib/posts.ts`)
3. **`app/[slug]/page.tsx`** — the service page template that uses (1) + (2) and is the template every other dynamic page is structurally based on

Everything else is variations on those three.

## What is intentionally not here

- **No CMS.** Content is markdown in git. Authoring happens in any editor. PR review is the workflow.
- **No client-side data fetching.** Every page is server-rendered or statically generated. No `useEffect` to fetch content.
- **No bespoke CSS.** Tailwind tokens and component utilities only. The single global stylesheet (`app/globals.css`) is small.
- **No third-party analytics SDK yet.** GA4 + Microsoft Clarity get added at launch via simple `<script>` tags. No heavyweight analytics package.
- **No dark mode.** Light theme only, by design (see `docs/DESIGN_SYSTEM.md`).
- **No carousels, no auto-play, no modals other than the mobile drawer.** Intentional simplicity for performance and accessibility.

## When something breaks

The single most useful debugging surface is `npm run dev`. Next.js prints clear error messages and the page's stack trace right in the terminal.

For schema validation, paste any URL into [Google's Rich Results Test](https://search.google.com/test/rich-results). If a schema block is malformed, the tool tells you why.

For mobile responsiveness, use Chrome DevTools device emulation at 360px (iPhone SE), 390px (iPhone 14), 768px (iPad portrait), 1024px (iPad landscape). Every page should render cleanly at all four.
